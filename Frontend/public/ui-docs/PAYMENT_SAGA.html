🧠 Advanced Recovery Mechanisms

🛡️ Transactional Outbox Deep-Dive

The "Dual Write Problem" states you cannot safely write to a DB and fire an API/MQ message—if step 2 crashes, data diverges. The Outbox solves this by placing the message intent into the exact same database. If the DB commits, the message is effectively "queued."

☠️ Poison Pills & DLQ Replays

If a downstream consumer errors continuously, RabbitMQ intercepts the exhaustion limits and automatically kicks the request to a Dead Letter Queue (*.dlq). A manual administrative endpoint /internal/dlq/replay executes recovery.

📊 State Transition Logic

CREATED → VERIFIED
VERIFIED → SUCCESS_PENDING
SUCCESS_PENDING → SUCCESS

Idempotent Event Consumption

Because RabbitMQ only guarantees At-Least-Once Delivery, network partitions may cause events to duplicate. Every downstream microservice uses an internal processed_events log mapped by the exact eventId (UUID). Standard INSERT ... ON CONFLICT DO NOTHING prevents duplicate business actions during retry storms.
