Overview & Core Philosophy

What is SkillSync? Why was it built this way?
🧠

1.0 — The "Absolute Beginner" Mental Model

▶

Before reading any instructions, you need to understand how the internet works using the Restaurant Analogy. This platform is split into four big pieces:

📱 1. The Frontend (React — You are here)

This is the Menu and the Dining Area. It's everything the user can see, click, and touch. It doesn't permanently save data—it just takes orders from the customer and looks pretty.

⚙️ 2. The API (The Network)

This is the Waiter. The Frontend gives an order to the Waiter ("Fetch me a list of mentors!"). The Waiter runs to the kitchen into the back, and brings the answer back to the table.

🖥 3. The Backend (Java Server)

This is the Kitchen & Chef. Customers are not allowed inside. The Chef receives the order from the Waiter, cooks the meal (processes the logic, checks security, charges money), and plates it up to send back.

🗄 4. The Database (PostgreSQL)

This is the Pantry/Fridge. The Chef (Backend) grabs ingredients (saved user data, passwords) from here. The Waiter (API) NEVER talks to the Fridge directly.

In this guide, we are ONLY talking about the Dining Area (React) and the Waiter (Axios/API). You are the interior designer of the dining area and the boss of the waiter.

🎯

1.1 — What is SkillSync?

▶

SkillSync is a mentor-learner marketplace platform — imagine Airbnb, but instead of renting houses, people rent knowledge! A student ("Learner") can find a teacher ("Mentor"), book a paid one-on-one session with them, join study groups, and track their learning progress. Mentors list what they can teach, set their available time slots, get paid real money through an Indian payment system called Razorpay, and build their reputation through reviews from students.

🍕
Think of it like ordering a pizza: You (Learner) browse a menu (Mentor list), pick what you want (a skill to learn), pay for it (Razorpay checkout), and get it delivered (attend the session). The restaurant owner (Admin) makes sure everything runs smoothly behind the scenes.

👩‍🎓 Learner (Student)

The person who wants to learn. They can search for mentors, book sessions (like booking a doctor's appointment), join study groups, and pay using Razorpay (an online payment system, like Google Pay).

🧑‍💼 Mentor (Teacher)

The person who teaches. They create a profile saying what they can teach, set when they're free (like a doctor's available slots), accept or reject session requests, and earn money for each session.

🛡 Admin (Manager)

The boss of the platform. They approve new mentor applications (making sure mentors are qualified), manage all users, manage skills/categories, and oversee the platform's money.

The project has two big parts:

🖥 Backend (the brain)

Built with Java + Spring Boot. This is the server — it stores data in a database, handles payments, sends emails, etc. Think of it as the kitchen in a restaurant — users never see it, but it does all the real work.

🎨 Frontend (the face)

Built with React + TypeScript + Tailwind CSS. This is what users see and click on — buttons, forms, pages. Think of it as the dining area of a restaurant. It's hosted on Vercel (a free hosting service) and talks to the backend at api.skillsync.mraks.dev.

🧱

1.2 — The Core Philosophy: Separation of Concerns

▶

The entire project follows one golden rule: each piece of code should do exactly ONE job. This is called "Separation of Concerns".

🏠
Real-world analogy: Think of a house. The kitchen is only for cooking. The bedroom is only for sleeping. The bathroom is only for bathing. You don't cook in the bathroom! Similarly, in our code, each file has ONE specific job, and it doesn't do anything else. This makes it easy to find and fix problems.
Pages — These are the full screens you see (like the Dashboard screen, the Login screen). A page is like a room in a house — it arranges furniture (components) inside it. Example: LearnerDashboardPage.tsx
Components — These are reusable building blocks — like LEGO bricks. A button, a sidebar menu, a popup message ("Toast") — these are all components. You build them once and reuse them on many pages.
Services — These are the "messengers" that talk to the backend server. When you click "Book Session", a service file sends that request to the server and brings back the response. Example: mentorService.ts handles all mentor-related requests.
Store Slices — This is the app's "memory". State = data the app needs to remember (like: "Is the user logged in? What's their name?"). A "slice" is one section of that memory. authSlice.ts remembers login info. sessionsSlice.ts remembers session data.
Hooks / Types / Utils — Helper tools. Hooks = reusable logic shortcuts (explained later). Types = descriptions of what data looks like (like a form template). Utils = small helper functions (like formatting a date).

1.5 — App Bootstrap Flow (main.tsx → screen)

▶

When a user opens skillsync.mraks.dev in their browser, the app starts up like a car engine — things happen in a specific order. Let's walk through each step:

🎁
What are "Providers"? A Provider is like a gift wrapper. It wraps around the entire app and gives every page inside it access to something useful. For example, ThemeProvider gives every page access to the dark/light theme. Without the wrapper, pages wouldn't know which theme to use. Think of it like electricity in a house — the main power supply (Provider) wraps the whole house so every room can use electricity.
main.tsx
→
Wraps app in: ThemeProvider (dark/light mode) → Redux Provider (app memory) → QueryClientProvider (data fetching) → GoogleOAuthProvider (Google login)
↓
App.tsx
→
Adds ToastProvider (popup messages) + ActionConfirmProvider ("Are you sure?" dialogs) + BrowserRouter (URL navigation)
↓
AuthLoader.tsx
→
Quietly checks: "Is the user already logged in?" by asking the server /api/auth/me. If yes, it restores their session so they don't have to log in again.
↓
AppRoutes.tsx
→
Looks at the URL (like /login or /learner) and decides which page to show. If the page needs login, it wraps it in a ProtectedRoute (a security guard).
↓
The page appears on screen! ✨
🧅
Why does the order matter? Think of it like layers of an onion. The outermost layer (ThemeProvider) wraps everything because even the login screen needs to know the theme. Redux wraps the app because the AuthLoader needs to save user data into it. The Router must wrap AuthLoader because AuthLoader checks the current URL. If you switch the order, things break — like putting on your shoes before your socks!

2.1 — React: Component-Based Architecture

▶

React is a tool that lets us build websites using a simple idea: break the screen into small, reusable pieces called "components".

🧱
Think of LEGO blocks: Instead of building an entire toy castle as one giant, unchangeable piece of plastic, you build it using small LEGO bricks (a door block, a window block, a roof block). In React, a "component" is a LEGO brick. You build a "Button" component once, and use it 50 times across your app. JSX is just the special language we use to describe what the LEGO brick looks like (it looks very similar to standard HTML).
Example: How ProtectedRoute.tsx works
// ProtectedRoute is a "guard" component
const ProtectedRoute = () => {
  // 1. Read auth state from Redux store
  const isAuthenticated = useSelector(
    (state: RootState) => state.auth.isAuthenticated
  );

  // 2. If not logged in → redirect to /login
  if (!isAuthenticated) {
    return <Navigate to="/login" replace />;
  }

  // 3. If logged in → render the actual page
  // <Outlet /> means: render whatever child route matches
  return <Outlet />;
};
🎓
For viva: "React components are like Lego bricks. ProtectedRoute is a security brick — if you're not authenticated, it redirects you. Outlet renders whatever child is matched — like a slot."
📐

2.2 — TypeScript: Why Types Matter

▶

TypeScript is an upgraded version of JavaScript. It adds "types", which means writing a strict rulebook for what your data should look like before the code even runs.

📋
Think of a bouncer at a club: Without TypeScript (normal JavaScript), any data can walk into your function. A number, a text string, or empty data — JavaScript lets them all in, and then the app crashes later when it realizes the data is wrong. TypeScript is the bouncer. You give it a list (an "interface") saying "Only let User data in, and User MUST have an email and an ID". If the wrong data tries to enter, TypeScript stops it with a red underline in your code editor before your app crashes.
types/index.ts — Your data contracts
// This tells TypeScript (and YOU) exactly what a User looks like
interface UserSummary {
  id: number;
  email: string;
  role: string;       // "ROLE_LEARNER" | "ROLE_MENTOR" | "ROLE_ADMIN"
  firstName: string;
  lastName: string;
  skills?: string[];  // ? means optional
  bio?: string;
}

// Payment has strict status values — no random strings allowed
interface Payment {
  status: 'CREATED' | 'VERIFIED' | 'SUCCESS' | 'FAILED';
}

Without TypeScript: you'd pass the wrong data, get undefined errors at runtime, crash in production. With TypeScript: errors show up in your editor before you run the code.

🎨

▶

Normally in web design (CSS), you invent a name like "my-button", and then in a separate file, you write 10 rules explaining what "my-button" looks like. Tailwind CSS gives you tiny, pre-made style rules that you stick directly onto your elements.

🎨
Think of ordering a sandwich: Traditional CSS is like giving the chef a paper that says "Make me The Special Sandwich", and the chef goes to a separate recipe book to know what that is. Tailwind is like pointing at the ingredients you want: "I want a sandwich with w-full (full width), bg-primary (blue background), and rounded-xl (rounded corners)". You build the design right where you ask for it.

2.4 — Functional Components + Hooks Pattern

▶

In modern React, every part of the screen is just a simple function. But sometimes a function needs superpowers — like memory to remember something the user typed. "Hooks" are special tools that give these simple functions superpowers.

🎣
Think of Hooks as utility belts: Imagine a plain function is just a normal worker. A Hook (like useState) gives that worker a notepad so they can remember things between tasks (this is called "state"). Another Hook (useEffect) gives them an alarm clock so they can do something at a specific time, like fetching data the moment the page opens (this is called a "side effect").
Hooks used in CheckoutPage.tsx
const CheckoutPage = () => {
  useState('card')        // local state: which payment method is selected
  useEffect(...)         // side effect: load Razorpay SDK on mount
  useLocation()         // React Router: read navigate state (mentorId, etc.)
  useNavigate()         // React Router: programmatic navigation
  useMutation(...)      // React Query: POST /api/payments/verify
  useRef()              // persist sessionId without re-render
  useToast()            // custom hook: show success/error toast
};

useRef is special — it's like a variable that survives re-renders but doesn't trigger a re-render when changed. Perfect for storing pendingSessionId during the payment flow.

🔄

2.5 — Custom Hooks: Reusability Pattern

▶

When you have a superpower recipe that you use often, you can package it into a custom hook. For example, fetching a list of mentors is done in many places. Instead of writing the fetch logic everywhere, we wrote useMentors().

📦
Think of an instant coffee pod: Instead of measuring the coffee beans, boiling the water, and mixing it every single time you want a cup (writing the API fetch code), you create a pod (a custom hook). Now, anytime a page needs mentors, it just uses the useMentors() pod and gets the data instantly!
hooks/apiHooks.ts — Reusable data fetching
// Define ONCE — use in many components
export const useMentors = () => {
  return useQuery({
    queryKey: ['mentors'],              // cache key
    queryFn: async () => {
      const { data } = await api.get('/api/mentors/search');
      return data;
    },
  });
};

// In any component, just:
const { data, isLoading, error } = useMentors();
// That's it! Caching, loading state, refetch — all handled

3.1 — What is Axios & Why Use It?

▶

Axios is a tool used by the frontend to send requests to the backend server. Think of it as a postman. While browsers have a built-in postman (called fetch()), Axios is a "smart postman" that can intercept letters before they are sent or directly after they arrive.

📮
What are Interceptors? Imagine telling your postman: "Before you deliver any letter from me, secretly attach my ID card to the envelope." That's a request interceptor. It automatically adds your security token to every API call. A response interceptor says: "If the backend replies with an 'Expired ID' error, pause the delivery, secretly go get me a new ID card, and then try delivering the letter again."
services/axios.ts — The central Axios instance
const api = axios.create({
  baseURL: API_BASE_URL,      // 'https://api.skillsync.mraks.dev'
  withCredentials: true,    // ← CRITICAL: sends HttpOnly cookies automatically
  headers: { 'Content-Type': 'application/json' },
});

withCredentials: true is the most important setting. Without it, the browser would never send the JWT cookies to the backend. It's your authentication handshake on every single request.

🔁

3.2 — Token Refresh with Queue-Based Retry Logic

▶

This is one of the smartest security features in your app. Your login "ticket" (access token) expires after 24 hours to keep hackers away. When it expires, the server gives an error (called a 401 error). Instead of rudely kicking the user out and forcing them to type their password again, the app secretly gets a new ticket in the background and retries the action.

🎟️
Think of a movie theater: You are watching a 3-part movie. After part 1, your ticket expires. The guard stops you. Instead of kicking you out into the rain, a helper (the app) tells the guard "Wait here", runs to the box office, gets a new ticket, hands it to you, and you walk straight in without even noticing the interruption. The "Queue Flow" below ensures that if 3 guards stop you at the exact same time, the helper only gets ONE new ticket, not three.

The Refresh Queue Flow

Request A hits 401
→
isRefreshing = false?
→
Set isRefreshing = true
→
POST /api/auth/refresh
↓ Meanwhile, 3 more requests return 401...
Requests B, C, D → 401
→
isRefreshing = true
→
Push to failedQueue[]
↓ Refresh succeeds...
processQueue(null)
→
All B, C, D replay automatically 🎉
↓ OR Refresh fails...
processQueue(error)
→
dispatch(logout()) → /login?reason=session_expired
⚠️
Why a queue? Without it, if 3 requests all fail simultaneously, you'd make 3 parallel refresh calls, creating race conditions and token chaos. The queue ensures only ONE refresh happens, then all waiters get replayed at once.
🗃

3.3 — React Query (TanStack Query): Server State Management

▶

React Query manages data that comes from the backend server. It handles everything relating to fetching, storing the fetched data, and keeping it up to date automatically.

🗄️
Think of a smart librarian: Normally, if you need a book (data from the server), you have to walk to the library, find it, and bring it back. If you need it again, you go back again. React Query is a smart librarian. When you ask for the "Mentors List", she gets it and makes a shiny copy for her front desk ("caching"). Next time you ask, she hands you the copy instantly. Then, in the background, she quietly checks if the original book changed ("background refetching"); if it did, she updates your copy without bothering you.

useQuery — for GET requests

const { data, isLoading, error } 
  = useQuery({
    queryKey: ['admin', 'stats'],
    queryFn: () => api.get('/api/admin/stats')
  });

Returns: loading state, data, error. Auto-cached for 5 minutes (see queryClient.ts)

useMutation — for POST/PUT/DELETE

const mutation = useMutation({
  mutationFn: (data) => 
    api.post('/api/payments/verify', data),
  onSuccess: () => navigate('/sessions'),
  onError: (err) => showToast({...})
});

Has: pending state, onSuccess, onError, onSettled callbacks

utils/queryClient.ts — Global defaults
new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,     // Cache for 5 minutes
      retry: 2,                       // Retry failed requests twice
      refetchOnWindowFocus: false,   // Don't refetch when tab regains focus
    },
  },
});

3.4 — Service Layer: mentorService.ts, sessionService.ts

▶

Instead of pages talking to the postman (Axios) directly, we created a service layer. This is a dedicated file that organizes all the API calls related to one topic, like mentors or payments.

📞
Think of receptionists: If everyone in a big company randomly yelled out the window to order supplies, it would be chaos. Instead, if someone needs something, they call the "Supply Receptionist" (the Service). The Receptionist knows exactly who to call, what forms to fill, and handles the actual ordering. If the actual supplier changes their phone number, you only update the Receptionist's notebook, not everyone in the building.
services/mentorService.ts — OOP service pattern
class MentorService {
  // Search with filters + pagination
  async getMentors(filters?, page = 0, size = 10) {
    const params = new URLSearchParams();
    if (filters?.skill) params.append('skill', filters.skill);
    // ... more filters ...
    const res = await api.get(`/api/mentors/search?${params}`);
    return res.data; // PaginatedResponse<MentorData>
  }

  async applyAsMentor(payload: MentorApplicationPayload) {
    const res = await api.post('/api/mentors/apply', payload);
    return res.data;
  }
}

// Exported as singleton — one instance, used everywhere
export default new MentorService();
🎓
For viva: "The service layer acts like a Repository pattern — it abstracts the API calls behind meaningful method names. If the API endpoint changes, I only update it in one place."

3.7 — Exponential Backoff for 429 & 503 Errors

▶

429 = Too Many Requests, 503 = Service Unavailable. Instead of failing immediately, the interceptor automatically retries with increasing delays.

Exponential backoff in response interceptor
const retryCount = originalRequest._retryCount || 0;

if ([429, 503].includes(error.response?.status) && retryCount < 3) {
  originalRequest._retryCount = retryCount + 1;
  
  // Wait: 1s, then 2s, then 4s (2^0 * 1000, 2^1 * 1000, 2^2 * 1000)
  const delay = Math.pow(2, retryCount) * 1000;
  await new Promise((res) => setTimeout(res, delay));
  
  return api(originalRequest); // retry the same request
}

Retry 1: wait 1s. Retry 2: wait 2s. Retry 3: wait 4s. After 3 retries, give up and reject.

4.1 — JWT Authentication: What is a JWT?

▶

JWT = JSON Web Token. It's an encrypted digital pass that contains your user info and a special stamp of approval. The server creates it, gives it to the app, and the app shows it to the server every time it asks for data.

🗝️
Think of a hotel keycard: When you check in at the reception (log in), they give you a keycard (the JWT). The keycard holds your room number and check-out date, but it's encrypted so you can't change it yourself to stay longer. Every time you want to enter a locked area like the pool or gym (make an API request), you tap the keycard to prove you are allowed in.

JWT Structure: 3 parts separated by dots

eyJhbGciOiJIUzI1NiJ9 . eyJzdWIiOiJ1c2VyQGV4YW1wbGUuY29tIiwicm9sZSI6IlJPTEVfTEVBUk5FUiIsImlhdCI6MTcxMDAwMDAwMH0 . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
RED = Header (algorithm)
YELLOW = Payload (userId, role, expiry)
GREEN = Signature (server's secret)

Your app uses HttpOnly cookies to store JWTs. This means JavaScript (including malicious scripts) cannot read the token — the browser handles sending it automatically. This protects against XSS attacks.

🍪

4.2 — HttpOnly Cookie vs localStorage: The Security Choice

▶

Once you get your "hotel keycard" (JWT), where do you keep it safe? There are two places in the browser: localStorage (a public pocket) and an HttpOnly Cookie (a locked vault).

🍪
Think of keeping your wallet safe: localStorage is like putting your wallet on the table in a restaurant. Anyone who sits at the table (any script running on the page, including bad ones) can take your cash. An HttpOnly Cookie is like putting your wallet in a safe that only the postman can open. You can't see the keycard, but whenever you send a letter (an API request) the postman secretly includes it for you.

4.3 — AuthLoader: Session Persistence on Page Refresh

▶

When you refresh your browser, the app's memory (Redux) gets completely wiped blank. But remember, your "keycard" is safely stored in the browser's vault (HttpOnly cookie). AuthLoader is the component that uses this keycard to rebuild your memory.

🔄
Think of waking up with amnesia: Every time you hit refresh, the app forgets everything. But the AuthLoader says, "Wait, let me take the keycard from the vault, send it to the server (/api/auth/me), and ask: 'Who is this person?'" The server replies with your name and role, and the app's memory is instantly restored.

4.4 — Role-Based Routing: ProtectedRoute + DashboardRedirect

▶

After a successful login, the app needs to send the user to the right place. A student goes to the Learner dashboard, while a teacher goes to the Mentor dashboard. The DashboardRedirect component acts as a traffic controller.

🚦
Think of a traffic cop: You show the traffic cop (DashboardRedirect) your ID badge (your role from Redux). If it says "Mentor", the cop points you to the Mentor lane. If it says "Admin", they point you to the VIP Admin lane.
routes/DashboardRedirect.tsx
const role = useSelector((state) => state.auth.role);

if (role === 'ROLE_ADMIN')  return <Navigate to="/admin"  />;
if (role === 'ROLE_MENTOR') return <Navigate to="/mentor" />;
                              return <Navigate to="/learner"/>;
💡
Authorization is enforced at two levels: Frontend (route guards) prevent wrong users from seeing pages, and Backend (Spring Security) rejects API calls from wrong roles. Never rely on frontend-only guards — a hacker can bypass client-side JS.

Frontend Security Implementations

XSS, CSRF, HTTPS, input validation, secure data handling
🛡

5.1 — XSS Prevention

▶

XSS (Cross-Site Scripting) happens when a bad guy sneaks evil code into your website (for example, by typing code into a comment box). React protects you by automatically sanitizing everything before it shows up on the screen.

🛡️
Think of entering an airport: An XSS attack is like someone trying to smuggle a weapon in their bag. React acts like the airport security scanner. Before any data reaches the screen, React checks it, removes anything dangerous (turns code into plain, harmless text), and only then lets it pass through.

❌ Dangerous (raw HTML)

// dangerouslySetInnerHTML
<div dangerouslySetInnerHTML=
  {{__html: userInput}} />
// NEVER do this with user data

✅ Safe (React auto-escapes)

// React escapes this safely
<div>{mentorName}</div>
// "<script>" becomes "&lt;script&gt;"

Additionally, storing tokens in HttpOnly cookies (not localStorage) means even if XSS somehow runs, it cannot steal the JWT.

🔒

5.2 — HTTPS Enforcement in Production

▶

HTTPS makes sure the connection between the user and the server is encrypted. Your app forces HTTPS so hackers on public Wi-Fi can't steal your users' hotel keycards (cookies).

🔒
Think of an armored truck: Sending data over HTTP (no 'S') is like sending cash in an open convertible — anyone on the highway can lean in and take it. HTTPS is like an armored, locked bank truck. Even if someone intercepts the truck, they can't see or steal what's inside.
axios.ts — HTTPS enforcement
const isProd = import.meta.env.PROD;

// If IP address slips through in prod, force HTTPS domain
if (isProd && configuredUrl.includes('35.153.59.2')) {
  configuredUrl = 'https://api.skillsync.mraks.dev';
}

HttpOnly cookies are also marked Secure on the backend — meaning the browser will never send them over HTTP. Your frontend on Vercel is HTTPS-only.

🚫

5.3 — CSRF Protection via SameSite Cookies

▶

CSRF (Cross-Site Request Forgery) = attacker tricks your browser into making requests to your app. Your backend sets cookies with SameSite=Strict, which means the browser only sends cookies for requests originating from your own domain.

💡
The frontend's role in CSRF protection is minimal — it's mostly a backend concern. But your choice of HttpOnly + SameSite cookies (vs localStorage + Authorization headers) is what provides this protection. If you used localStorage, CSRF would be a real threat.
🔑

5.4 — Environment Variables: Keeping Secrets out of Code

Sensitive config (API URLs, OAuth client IDs) lives in .env files, not hardcoded in source code. Vite exposes env vars prefixed with VITE_ to the frontend.

State Management Deep-Dive

Redux Toolkit + React Query — two different jobs, two different tools
🏗

7.1 — Why Two State Libraries? Redux + React Query

▶

This is the most common viva question. Redux Toolkit and React Query are both "memory managers", but they remember entirely different things:

🧠
Think of the app's brain having two sides:
Redux (Short-term personal memory): Remembers things happening right now on your screen. "Is the sidebar open?", "Am I in dark mode?", "Who is logged in?". It forgets everything when you refresh the page.
React Query (Library card memory): Remembers facts from the outside world (the server). "What's the list of mentors?", "What are my upcoming sessions?". It automatically checks if facts have changed and updates them.

🔴 Redux Toolkit

Client State — data that lives in the browser, not the server

Examples: Is user logged in? What role? Is sidebar open? Which theme?

Persists while React is running. Lost on refresh (that's why AuthLoader exists).

🟢 React Query

Server State — data that comes from the backend API

Examples: List of mentors, session data, notifications, admin stats

Automatically cached, refetched in background, synced with server

🎓
"I use Redux for auth state because it needs to be synchronous and available everywhere before any API call happens. I use React Query for API data because it gives me caching, loading states, and background refetch out of the box."
📊

7.3 — Redux Toolkit: createSlice Explained

▶

createSlice is just a shortcut for setting up a section of memory. It automatically creates the actions (the orders you give) and the reducer (the worker who writes down the new memory).

🍰
Think of a pizza slice: The whole pizza is the "Root Store" (all the app's memory). A "slice" is just one piece of that memory — like the "auth slice" that only remembers login info. createSlice is the pizza cutter. It neatly packages everything you need to manage that specific slice without messing up the rest of the pizza.
authSlice.ts — How createSlice works
const authSlice = createSlice({
  name: 'auth',              // prefix for action types
  initialState,              // starting values
  reducers: {
    setCredentials(state, action) {
      // RTK uses Immer — you can "mutate" state directly
      // Immer makes an immutable copy behind the scenes
      state.user = action.payload.user;
      state.isAuthenticated = true;
    },
    logout(state) {
      state.user = null;
      state.isAuthenticated = false;
    }
  }
});

// RTK auto-generates: auth/setCredentials, auth/logout
export const { setCredentials, logout } = authSlice.actions;

Using it in a component:

Reading and writing Redux state
// READ (useSelector)
const user = useSelector((state: RootState) => state.auth.user);

// WRITE (useDispatch)
const dispatch = useDispatch();
dispatch(logout());  // triggers the logout reducer

13.2 — useMemo: Preventing Expensive Recalculations

▶

useMemo is a React hook that literally tells the app to "memorize" the answer to a hard math problem. It only does the math again if the numbers change.

🧮
Think of doing long division: If someone asks you "what is 9243 ÷ 3?", you do the hard math and say "3081". If they ask you the exact same question two seconds later, you don't do the math again! You just remember the answer. useMemo is how we teach the app to remember answers.
ThemeContext.tsx — useMemo for context value
// Without useMemo: new object created on EVERY render
// → all consumers re-render unnecessarily

// With useMemo: same object reference until [theme] changes
const value = useMemo(
  () => ({ theme, isDark: theme === 'dark', setTheme, toggleTheme }),
  [theme]  // ← only recalculate when theme changes
);
🔂

13.3 — useCallback: Preventing Function Recreation

▶

useCallback is similar to useMemo, but instead of memorizing an answer, it memorizes a set of instructions (a function).

📜
Think of a recipe card: Every time the app redraws the screen, it throws away all old instructions and writes new ones from scratch. This takes time! useCallback is like laminating a recipe card. The app uses the same laminated card every time instead of writing a new one, which makes the app run much faster.
Toast.tsx — useCallback for showToast
// Without useCallback: showToast is a new function every render
// → components using useToast() would re-render unnecessarily

const showToast = useCallback(({ message, type }) => {
  setToastData({ message, type });
  setTimeout(() => setToastData(null), 3000);
}, []); // ← empty deps = created ONCE, never changes

🔄 REST Polling (current)

Client asks server every N seconds: "Any new notifications?" Server responds with count. Simple but wastes bandwidth on empty responses.

Like kids on a road trip: "Are we there yet? No. Are we there yet? No." It's annoying and wastes energy.

⚡ WebSocket (future)

One persistent TCP connection. Server pushes notification instantly when it occurs. Zero polling overhead.

Like a smart intercom: The driver says "Stop asking. I will leave this intercom open, and the second we arrive, I will announce it." It's instant.

The Promise.all pattern in clearAllNotifications() is a concurrency technique — all delete requests fire simultaneously instead of sequentially, making bulk deletion much faster.

15.2 — Rollback Pattern: Why useRef?

▶

The pendingSessionIdRef uses useRef instead of useState deliberately. Here's why:

❌ useState would cause

Setting state causes a re-render. In the middle of the payment flow, re-renders could break the Razorpay modal or reset loading states unexpectedly.

✅ useRef is perfect

ref.current stores a value that persists across renders but never triggers a re-render when changed. Safe to update during async operations.

Rollback mechanism
const pendingSessionIdRef = useRef<number | null>(null);

// After creating session: store its ID for potential rollback
pendingSessionIdRef.current = sessionId;

// If user closes modal, payment fails, or verify fails:
const rollbackPendingSession = async () => {
  const sessionId = pendingSessionIdRef.current;
  if (!sessionId) return; // nothing to rollback
  
  try {
    await api.put(`/api/sessions/${sessionId}/cancel`, ...);
  } catch { /* best-effort — don't crash on rollback failure */ }
  finally { pendingSessionIdRef.current = null; }
};

15.4 — Payment Signature Verification (Security)

▶

After payment, Razorpay sends 3 values. The backend verifies the HMAC signature by recomputing it with the secret key. This prevents anyone from faking a "payment success".

🔏
Think of a wax seal on a royal letter: After a user pays, Razorpay hands them a receipt and says "Give this back to SkillSync." But what if a hacker changes the receipt from $5 to $500? To prevent this, Razorpay stamps the receipt with an unsolvable mathematical wax seal (the Signature). When your server gets the receipt, it checks if the seal is perfectly intact. If the math matches, we know the receipt is 100% genuine and hasn't been tampered with.
ValueWho Creates ItWhat It's For
razorpay_order_idRazorpayReference to the order created in step 2
razorpay_payment_idRazorpayUnique ID of this specific payment transaction
razorpay_signatureRazorpayHMAC-SHA256(orderId + "|" + paymentId, secret)
🎓
"The signature verification happens on the backend, not the frontend. The frontend just collects the three values from Razorpay's callback and sends them to our Spring Boot service. If someone tampered with orderId or paymentId, the HMAC would not match and the verification would fail."

Communication & Challenge Resolution

How to explain design decisions in the viva
🗣

20.1 — Common Viva Questions & Ideal Answers

▶
❓

Why did you use Redux when you already have React Query?

▶

"They serve different purposes. React Query manages server state — data from the API that can go stale, needs caching, and gets refetched. Redux manages client state — things like whether the user is authenticated, their role, and UI state like sidebar visibility. Auth state needs to be synchronously available everywhere before any API call happens, which is why Redux is the right choice for it. React Query would require an async query which introduces race conditions during initial load."

❓

Why HttpOnly cookies and not localStorage for JWT?

▶

"localStorage is accessible by any JavaScript running on the page. If there's an XSS vulnerability — even in a third-party script — the token can be stolen. HttpOnly cookies are inaccessible to JavaScript by design; only the browser sends them. Combined with the Secure flag and SameSite=Strict, this protects against both XSS token theft and CSRF attacks. The original code had localStorage, but it was explicitly removed as a security improvement."

❓

What happens if the user closes the browser during payment?

▶

"If the user closes the browser after the session is created but before payment completes, the session stays in REQUESTED state in the database. When the user returns and the page unmounts, the useEffect cleanup doesn't run. This is a known limitation. The ideal solution is a server-side scheduled job (cron job) that marks sessions as CANCELLED if they've been in REQUESTED state for more than, say, 15 minutes without a corresponding verified payment. The frontend handles the dismiss case (Razorpay modal's ondismiss callback) by calling the rollback API."

❓

Why does AuthLoader check /api/auth/me on every route change?

▶

"It only calls /api/auth/me if the Redux user is null — meaning on page refresh or first load where Redux state is empty. The user object in Redux is a dependency in the useEffect, so once it's loaded, the effect won't re-run unless location.pathname changes AND user is still null. This prevents redundant /me calls on normal navigation. The only cost is one API call on hard refresh, which is acceptable and necessary for session persistence."

21.2 — The Mounted Flag Anti-Pattern Prevention

▶

In AuthLoader, there's an advanced pattern that prevents memory leaks and React warnings:

AuthLoader.tsx — Mounted flag
useEffect(() => {
  let mounted = true; // track if component is still in DOM

  const initAuth = async () => {
    // ... async operations ...
    
    // Only update state if component is still mounted
    // (user might navigate away during async operation)
    if (mounted) setLoading(false);
    if (mounted && data) dispatch(setCredentials(...));
  };

  initAuth();

  // Cleanup: set mounted = false when component unmounts
  return () => { mounted = false; };
}, []);
⚠️
Without this pattern: if the user navigates away before the async /api/auth/me call completes, React would try to setState on an unmounted component, causing a memory leak and the famous "Can't perform a React state update on an unmounted component" warning.
