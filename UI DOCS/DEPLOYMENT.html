SkillSync Deployment Architecture

Current real deployment path with manual release operations and EC2 runtime steps.

1. Runtime Topology

  • Frontend hosted separately (domain-driven static deployment).
  • Backend microservices run in Docker Compose on EC2.
  • Data and infra containers: PostgreSQL, Redis, RabbitMQ, Zipkin, Prometheus, Grafana, Loki.

2. Why manual deploy now

  • CI hosted minutes quota exhausted.
  • Manual deploy is currently the stable fallback.
  • Same production architecture, only release trigger changed.

3. Manual Deployment Runbook

  1. On dev machine: push source to GitHub.
  2. Build and push changed backend image(s) to Docker Hub.
  3. SSH into EC2 and run git pull.
  4. Run docker compose pull in Backend directory.
  5. Run docker compose up -d --remove-orphans.
  6. Verify with docker compose ps and service logs.

4. Validation Checks

  • Gateway health endpoint is up.
  • User service and auth service healthy.
  • Frontend can hit auth and profile APIs.
  • No restart loops in container logs.

5. Rollback Strategy

  • Retag or reference previous known-good image.
  • Pull old image and restart only impacted service.
  • Keep compose and environment backups.

6. Security Notes

  • Secrets remain in EC2 environment files, never in repo.
  • Keep public ports minimal.
  • Rotate exposed credentials and tokens immediately.
Important: For backend code changes, git pull alone is not enough. You must also pull updated Docker images and restart containers.
